## In-retro production bug (found by using the feature)

Luca ran a real sync with an expired cookie: no attention banner appeared. Root cause (probed live against PSN, not assumed): **PSN answers an expired cookie with HTTP 200 + a GraphQL `"Access denied! You need to be authorized…"` error — never 401/403.** The provider's only banner-lighting branch keyed off 401/403, so the real path threw a generic error → 502 → "try again later" toast → `markPsnAuthExpired` never ran → banner stayed dark.

The entire recovery chain (flag → banner → instructions → cookie-save-clears) was correct and tested — with *simulated* 401s. The trigger condition was reverse-engineered from HTTP convention instead of observed PSN behavior. 4.1's own residual-risk note called it ("live PSN 401 path ships unexercised until 4.2"); it shipped unexercised, period.

Fix: one branch in the adapter mapping the 200+denial payload to `PsnAuthError` (AR-5 encapsulation held), plus a hazard test using the **captured real payload** as fixture. Verified end-to-end in the live browser — banner now lights. Same shape as Epic 3's biggest discovery: **real usage catches what a PSN-stubbing test pyramid structurally cannot — because the stub encodes the assumption under test.**

## Challenges

1. **Assumed wire contracts.** The epic's one production bug and the WEBMAF gap share a root: PSN was modeled from convention + the Python script's happy path, never probed for failure modes. The test pyramid then verified the assumption, not reality.
2. **High-severity cluster.** 6 highs after a zero-high Epic 3 — all in the sync core (matching, dedup, race-safe inserts). Caught pre-merge. The lesson isn't "quality dropped": sync-shaped code (external data → automated writes) is a higher risk class than UI stories, and review effort concentrated there correctly.
3. **Spec-then-reverse.** FR-9's claims-skip was specced, built, tested — then reversed by product decision within a day, discovered by using the real shelf. Second epic running where the significant discovery came from Luca using the feature after the spec froze. Reframed by Alice as the system working: solo-dev BMAD can't focus-group upfront; cheap same-day reversals are the substitute, and append-only + migration discipline kept the cost small.

## Significant Discovery — the follow-up-review gate leaked (again)

All three 4.x specs carry `followup_review_recommended: true`. Epic 1's retro built a gate for exactly this (a story can't reach done with the flag unresolved) — implemented as `.bmad-loop/policy.toml` `[review] trigger = "always"`. But **bmad-loop was retired on Windows** (the unattended-epic experiment pivoted to Claude Code driving `bmad-dev-auto`). The current flow runs the inline review subagent (the Review Triage Logs — those caught the 34 patches) but **nothing reads `followup_review_recommended` anymore**; the independent second-opinion session lived in the retired tool. The stamps sat unconsumed.

Same failure shape as Story 1.5 in the Epic 1 retro: a review signal raised, silently dropped, named hazard ships. And 4.1's flag came with the exact residual note describing today's bug. **Meta-lesson (Dana): process rules welded to a specific tool die with the tool.** The hazard-test rule survived because it lives in `_bmad/custom/bmad-dev-auto.toml` (skill customization); the follow-up-review rule died because it lived in bmad-loop's policy file.

**Resolution this session:** ran the independent review over the epic-4 branch (8 finder angles + verify) — 0 correctness bugs; 3 quality findings, all addressed (2 `ponytail:` ceiling comments, 1 ledgered). The three stamps are consumed.

## Key takeaways

1. **Real usage is the test the pyramid can't write.** Two epics running, the significant discovery came from Luca using the feature after the spec froze — because a provider stub encodes the very assumption that needs testing. Probe the real endpoint's failure modes; fixture from captured payloads.
2. **Process rules must live where the current toolchain enforces them.** The follow-up-review gate leaked because it was welded to bmad-loop (retired); the hazard-test rule survived because it lives in the skill customization. Re-home the gate.
3. **Cheap reversals beat expensive upfront certainty for solo-dev BMAD.** FR-9 flipped in a day at low cost because append-only semantics and migration discipline made it cheap. That's the system working, not a planning miss.
4. **Sync-shaped code is a distinct risk class.** 6 highs, all in external-data→automated-writes core, all caught pre-merge. Concentrate review effort there; it's not UI-story risk.
