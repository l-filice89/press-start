## 2. What went well

**The spike killed a story, and that is a win.** Story 9.4 (wishlist sync) did not get built, half-built, or ground on. Two spikes — 9.1 against live PSN, 9.1c through an authorised signed-in browser session — established that the wishlist read is **server-side-rendered**: `__NEXT_DATA__` carries the data, no client-side `storeRetrieveWishlist` request ever fires, and the operation is genuinely absent from the client-reachable persisted allowlist. The investigation went as far as reconstructing Apollo's own hashing pipeline in-page and **validating the recipe exact** against a control query the app *does* execute client-side (`getCartItemCount`, hash reproduced byte-for-byte) before concluding the wishlist doc simply is not registered. That is a negative result proven to the same standard as a positive one, and it retired a story with evidence instead of opinion.

**The epic was re-sequenced by its own spike, correctly.** S-1 found the `pdccws_p` cookie 401'd by the trophy host and the NPSSO bearer serving *both* trophies and the existing library sync — a superset. That promoted an auth swap out of "Deferred" and put it in front of every trophy story. The unattended loop **halted rather than inventing a sequencing decision**, which is exactly what it should do when the shape of the epic changes underneath it.

**PROBE-BEFORE-YOU-MAP earned its keep three separate times.** Each probe overturned an assumption a competent engineer would have shipped:

1. The cookie authorizes trophies → **401**. (Caught before a line of trophy code existed.)
2. The trophy *list* carries earned dates → **it does not**. (Forced the per-title detail endpoint into 9.3's design.)
3. `npServiceName` is `trophy2` → **94 of 137 titles are `trophy`**. (See below — this one nearly shipped.)

**The architecture held under a credential transplant.** 9.1b swapped the auth mechanism end-to-end — key, seed secret, getter, UI field, help copy, banner variant — across Epic 4's library sync and Epic 5's catalog paths, and both stayed green. The AR-5 rule that all PSN auth lives *inside* `PsnProvider` is why that was a contained change and not an epic-wide excavation. The encapsulation test (`psn-encapsulation.test.ts`) was updated to fence the new `ca.account.sony.com` URLs into the adapter, so the containment is enforced rather than merely intended.

**The cookie path was deleted, not left beside the bearer.** No parallel credential, no fallback, no dead weight — and migration 0006 reaches into a deployed D1 to delete the stale `psn_cookie` row at rest. A rename that leaves the old row behind is a rename that hasn't happened.

**A deliberate deviation from the epic AC was argued, not silently taken.** The epic said the adapter should "refresh the bearer from the offline refresh token when the cached bearer expires." 9.1b noted there *is* no bearer cache to expire — a Worker invocation is stateless, and caching the bearer across invocations means persisting a second derived credential at rest to save two subrequests. The NPSSO is the durable credential; the exchange is memoized per provider instance. The user-visible promise of the AC holds exactly. That is the right way to disagree with a spec: in writing, with the reasoning, in the story.

## 3. What did not go well — the spec-quality gap

**The finding of this retrospective, stated once:**

> Every safeguard this project has installed lives **downstream** of the spec. PROBE-BEFORE-YOU-MAP, the DEGENERATE-RESPONSE GUARD, the HAZARD-TEST rule, FOLLOW-UP-REVIEW AUTO-FORCE ON HIGH — all of them assume the spec is honest and catch the *code* when it isn't. Epic 9's HIGHs say that assumption is no longer safe: **three of the eight HIGHs were the spec being confidently wrong, and the code faithfully implementing it.**

The three:

**1. `npServiceName: 'trophy2'` — a constant asserted from one sample.** Story 9.3's spec pinned the trophy-detail service name from a single PS5 observation. A live probe, run *only because a reviewer challenged it*, measured the real distribution: **94 of the account's 137 titles are `trophy`** (PS3/PS4/Vita), 43 are `trophy2` (PS5) — and the wrong name answers 404, which the backfill would have reported to the user as *"PlayStation no longer has trophy data for this title."* Had it shipped, **most of a real user's platinums would have been silently skipped as delisted, into a write-once column, permanently unrecoverable.** PROBE-BEFORE-YOU-MAP was followed — a probe *was* run — but probing one happy-path response tells you the **shape** of a record and says nothing about the **distribution** across records.

**2. The 9.2 subrequest budget was measured against a mock that could not see the real cost.** The spec counted 4 external fetches and declared the run comfortably inside the 50-subrequest ceiling. It ignored the ~137 sequential D1 `UPDATE`s the write loop issued — **D1 binding calls count toward the Workers subrequest limit**, and no test could see it because the mock doesn't charge for them. Every suite was green. Fixed with `setTrophyCountsBatch` (chunks of 50 via `db.batch()`): 137 writes now cost 3 batch calls.

**3. The 9.3 chunk budget repeated the shape.** The chunk size (20 per request) didn't count the auth middleware's own D1 reads. Lowered to 15 with the arithmetic restated honestly.

**Charlie's framing, which the team accepted:** *"It's not that the specs are sloppy. It's that a spec is allowed to state a measurement — a constant, a distribution, a budget — with no evidence and no marker saying it's unverified. The reviewer has to be personally suspicious to catch it. That's luck, not process."*

**Luca's call:** put the safeguards on the specs. This epic's review pass was excellent and the code came out hardened, but "the reviewer happened to be suspicious" does not scale, and the failure mode it guards is silent, permanent data loss.

### The other HIGHs (all patched, all data-integrity, none cosmetic)

- **PS4 and PS5 trophy sets silently overwrote each other** (9.2). PSN lists them as two entries ("Hades" / "Hades Trophies") that normalize to the same game; the write loop kept whichever PSN returned last — an abandoned 3% PS4 run could replace a 100% PS5 platinum, with no attention item. Found independently by *both* reviewers. Now collapsed by match key before any write (most earned wins).
- **The library side of the join stripped " Trophies" off library titles** (9.2) — the exact thing `core/trophy.ts` documents as forbidden. Now keys on the stored `title_normalized` column the library sync already matches on, so the two syncs cannot drift.
- **One permanently-failing title stranded every candidate behind it, forever** (9.3). Any non-auth error aborted the chunk; candidates are ordered by `game_id` and filled rows drop out of the set, so every re-run marched into the same bad title and died there — nothing with a higher id could *ever* be backfilled.
- **A missing timezone silently stamped every date in UTC** (9.3). `platinum_on` is write-once, so an evening platinum misdated by a day was permanent. The run now refuses (409) without a captured timezone.
- **Mid-chunk failures reported rows already written as if nothing had happened** (9.3). A token expiring on candidate 11 of 20 left 10 dates written but dropped them from the report — and write-once means they could never be re-reported.
- **Denial vs. outage was collapsed** (9.1b). Both OAuth legs mapped *any* non-code answer — a Sony 5xx, a 429, a 403 WAF challenge — to `PsnAuthError`, which flags the credential expired and tells the user to re-paste a perfectly valid token, a remedy that cannot work.

Common thread across all six: **they are all "the sync wrote the wrong thing, quietly, into a column that can never be corrected."** Not crashes. Not 500s. Silent, permanent, plausible-looking wrong data. That is the failure class this codebase's write-once/append-only design is *most* exposed to, and it is exactly where the review effort should keep going.

## 9. Key takeaways

1. **Probing one response tells you the shape of a record; it never tells you the distribution across records.** The `trophy2` near-miss is the most expensive lesson of the epic, and it hid *behind* a rule the team was already following correctly.
2. **A budget a mock cannot observe is not a budget.** Two stories in one epic claimed a subrequest ceiling while ignoring the D1 calls that actually consume it, and every test stayed green.
3. **Auto-forcing an independent review on any HIGH is the highest-leverage rule this project has.** It fired three times in Epic 9 and found a further real defect every time. Epic 6 wrote it; Epic 9 proved it.
4. **A story killed on evidence is a story delivered.** 9.4 cost two spikes and produced no code, and that is the correct outcome — the alternative was grinding indefinitely on an endpoint Sony does not expose to clients at all.
5. **The safeguards had all been aimed at the code. The spec was the unguarded surface.** Epic 9's two new rules are the first to bind the author of the spec rather than the writer of the code.
