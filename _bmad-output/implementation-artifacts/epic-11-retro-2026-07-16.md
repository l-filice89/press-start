## What went well

- **Speed with cleanliness (Luca's headline).** One day from account lock to full credentialed-surface removal, at the lowest defect severity of any epic. The speed was the accumulated process paying out: same-morning correct-course, a tight epic context doc ("forward-only excision, not a git rollback"), and eleven epics of standing rules letting dev-auto run three removal stories back-to-back with zero HIGHs.
- **Surgical non-deletion.** A removal epic's biggest risk is over-deletion. Boundaries held everywhere: `platinum_on` ≠ `trophy_earned_platinum`, PlatinumTrophy badge and manual milestone/ownership flows byte-for-byte untouched (proven by unmodified Epic 2/6.4 suites + byte-identical survivor asserts in the migration hazard tests), `sync-reconcile.ts` kept while production-dead because a surviving test pins it.
- **Reviewers verified instead of assuming.** 11.2's top review finding ("deployed Wrangler secret still holds the token") was **verified false live** via `wrangler secret list` and rejected with evidence — as valuable as a patch.
- **Hazard tests of the right shape.** Severed routes pinned to 404; migration 0011's test asserts the exact post-migration column list with non-NULL byte-identical survivors (hardened from a weak subset check by review); the migration-test isolation fix (`slice(0, target + 1)`) caught a real regression (0011 breaking 0010's assert) in the same session it shipped.
- **The epic left a permanent invariant, not a point-in-time grep** — the CI grep-clean guard pins "no credential ever reaches the wire" forever.

## Challenges / lessons

1. **The real finding is upstream of the epic.** Epic 11 is the cleanup invoice for a risk decision made in Epics 4/9: a credentialed sync impersonating the PSN Android app against undocumented endpoints, with the risk known, researched (spike 9.1 probed the credential's authorization scope), stated in plain text in the specs — and never gated. Luca's verdict: **in hindsight a bad decision that got a cheap lesson** — the features had value, but the cost outweighed it. The process had a manual-approval gate for irreversible *data* actions (destructive migrations) and none for irreversible *external* risk. → Action item 1 (rule shipped this session).
2. **The 11.3 implementation agent died mid-run** (session limit, ~90% complete) on the story carrying the destructive migration; the orchestrator completed COVERAGE mapping, ledger notes, the e2e absence pin, and the 0010-test regression. Recovery worked; second infrastructure-limit incident shaping a run. Watch, no action yet.
3. **Stale action items in sprint-status, second retro in a row** (Epic 9's items at the Epic 7 retro; Epic 7's items 4/5 today). Items completing *outside* a story (merges, deploys, kickoffs) have no mechanism that flips them. Accepted: the retro audit is the mechanism, and it has caught 100% at zero added process cost. No new machinery.

## Key takeaways

1. **A known risk without a named checkpoint is an unflagged bet.** The ToS risk was researched and written down, and still never had a moment that forced accept/reject. Irreversible external risk now gets the same human pause as irreversible data loss.
2. **Removal epics need surgical boundaries more than addition epics need specs.** Every deleted symbol is a bet that nothing kept needed it; the Always/Never constraint blocks and byte-identical survivor asserts are why this one shipped clean.
3. **The rule pipeline compounds.** Epic 7's delegation rule fired on its first epic and closed the exact gap class that created it; seventh consecutive clean follow-through on retro commitments.
4. **Verify review findings against reality before patching.** The epic's top-severity review claim was disproven live in one command — evidence-based rejection kept a false fix out of the diff.
5. **Speed is downstream of accumulated discipline.** One-day emergency response at record-low defect severity was not heroics; it was eleven epics of standing rules doing their job under pressure.
