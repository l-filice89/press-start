## Intent

**Problem:** Signing into my own app means waiting for an email. Google is the account I already have; one tap should do it. (B1a — the one publication blocker that is single-tenant-safe and gates nothing.)

**Approach:** Add Google as a better-auth **social provider alongside** magic link — neither replaces the other. The single-user `isAllowedEmail` allowlist stays exactly as it is and now also governs the OAuth path: a non-allowlisted Google account is rejected before a user row is written, and the rejection is stated on the login screen. No schema migration; the app stays single-tenant (dropping the allowlist is Story 8.2).

## Boundaries & Constraints

**Always:**
- Both paths work: the magic-link form and a Google button both render on the login screen; magic link is untouched.
- The allowlist gate rejects a non-`AUTH_ALLOWED_EMAIL` OAuth callback. Reject **before** a `user` row is created — a blocked sign-in must leave no user/account residue in D1, the same "no database residue" rule `routes/auth.ts` already applies to magic link.
- The rejection is visible: the user lands back on the login screen with a plain statement of what happened, never a blank screen or a silent bounce.
- The Google client secret is a Worker secret (`wrangler secret put` / `.dev.vars`) — never in `wrangler.jsonc`, never in the repo. The client **id** may be a secret too; keep both in the same place rather than splitting them across config files.
- An unset/empty allowlist means nobody, on the OAuth path exactly as on the magic-link path (fail closed).
- Existing token system only — no new palette, no icon dependency.

**Block If:**
- The allowlist cannot be enforced on the OAuth path without a schema change or without touching the magic-link flow.

**Never:**
- No schema migration, no registration/self-signup, no roles, no sharing, no per-user data scoping (all of that is Stories 8.0/8.2+).
- Do not weaken, replace or bypass `isAllowedEmail`.
- Do not add a Google/OAuth npm dependency — better-auth ships the provider.

## I/O & Edge-Case Matrix

| Scenario | Input / State | Expected Output / Behavior | Error Handling |
|----------|--------------|---------------------------|----------------|
| Allowed Google account | OAuth callback for `AUTH_ALLOWED_EMAIL` | Session established, redirect to `/`, shelf loads — same session shape as a magic-link sign-in | No error expected |
| Non-allowlisted Google account | OAuth callback for any other email | No `user` (and no `account`) row is written; redirect back to `/?error=ACCESS_DENIED` | Login screen states the account isn't allowed to sign in |
| Allowlist unset/empty | `AUTH_ALLOWED_EMAIL` absent, any OAuth callback | Rejected — fail closed, identical to the magic-link gate | Same ACCESS_DENIED rejection |
| Google creds not configured | `GOOGLE_CLIENT_ID`/`SECRET` unset (dev, e2e) | Magic link keeps working; the Google button surfaces a plain "couldn't start" error rather than a blank bounce | The provider is simply not registered; the button's request fails and the login error line says so |
| Stale allowlist | A `user` row exists whose email is no longer allowlisted (allowlist changed after signup) | Its session is refused at the protected-route seam (401), on every route | Defense-in-depth; the sign-in path's gate is the primary |

</intent-contract>

## Design Notes

The allowlist cannot be enforced at the Hono route boundary on the OAuth path the way it is for magic link: the email only exists **after** better-auth exchanges the authorization code. better-auth's own callback (`api/routes/callback.mjs`) wraps `handleOAuthUserInfo` — which runs the `user.create.before` database hook — in a `try/catch` that turns an `APIError` with a `code` into `redirectOnError(errorURL, code)`. So throwing from the hook is exactly the "no database residue + a machine-readable error on the login screen" behavior the magic-link pre-gate already gives us:

```ts
databaseHooks: {
  user: { create: { before: async (u) => {
    if (!isAllowedEmail(u.email, env)) {
      throw new APIError('FORBIDDEN', { code: 'ACCESS_DENIED', message: '…' });
    }
  } } },
}
```

`Login.tsx` already reads and consumes `?error=`, so the rejection needs one new branch, not new plumbing.
