- **The forced follow-up round earned its status again.** Two real HIGHs found post-"done" (bad-region guard re-firing store fetches per shelf GET; genre add/remove actor-only bump), plus the `windowOf` root-cause fix.

## Challenges / recurring patterns

1. **The stale-304 family — five findings, one class.** Shared fact written, only the actor's library version bumped → co-trackers 304'd against stale data forever. Both 8.6 HIGHs, the follow-up's new HIGH, two follow-up MEDs. Only became possible when 8.3 turned reads into derivations. Rule extracted in-epic: *test the bypass path per writer category, not just the 304 refusal.*
2. **The userless cron state machine took two passes.** All three 8.4 HIGHs clustered on the guard/ledger interaction; the backoff bug resurrected on the `bad-region` branch in follow-up; `windowOf` (calendar month vs 15–28 cron window) was the shared root of two MEDs.
3. **Operational/temporal concerns had no agent owner — Luca was the missing reviewer.** The scores-cron split, the all-users failure flag, the orphaned `findOldestUser` delegation, and the zero-headroom budget relief were all raised by Luca post-epic. Every item was ledgered (visible); none was prioritized (no owner, no trigger). Reviews audit diffs; nobody audits time. → Action item 1.
4. **Oversized stories carried the most HIGHs.** 8.3 and 8.4 both shipped `warnings: ['oversized']` and together account for 4 of the 17 inline HIGHs plus most of the follow-up findings.

## Key takeaways

1. **Research before design before code compounds.** The capacity numbers gave the gate something falsifiable; the gate gave reviews a contract to enforce. Each stage handed the next one evidence, not intent.
2. **Derive, don't copy.** Per-user answers as SQL derivations over shared snapshots killed the write cliff and made coexistence structural — the epic's exportable architecture lesson.
3. **Reviews audit diffs; nobody audits time.** The one systematic gap was operational: cadence, recovery, month-three behavior. The new sweep rule gives that perspective an owner and a trigger.
4. **A new read architecture mints a new defect class.** ETag/304 derivation created the stale-bump family within the same epic that introduced it — when the read model changes, enumerate the writers, not just the readers.
5. **Ledger visibility is not ledger priority.** Every post-epic fix was already written down; what was missing was a mandated triage moment. Writing things down is half the system.
