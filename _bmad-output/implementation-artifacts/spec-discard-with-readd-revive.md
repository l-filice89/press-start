## Intent

**Problem:** A game added by mistake (typically a name-only add-by-name typo) has no way out of the library — `Dropped` is a play-status, not "this shouldn't be here." A prior decision (sprint-status 2026-07-10) rejected discard because hard-delete resyncs and irreversible soft-delete forced revival. This flips that objection: re-add-revival is the wanted behavior.

**Approach:** A `discarded` boolean tombstone on `game_tracking` (the "tracking-level archived" path that decision left open). Discarding hides the game from every library surface but keeps the row; sync must not re-own it; re-adding the same name clears the flag (revive). No browse-list — re-add is the only discovery/revive path. Also fix the stale `.dev.vars.example` IGDB block (G3).

## Boundaries & Constraints

**Always:** Every discard/revive write is user-scoped (AD-13). Discard NEVER deletes the `game` or `game_tracking` row — it only sets the flag (the tombstone is what blocks re-add duplication and survives sync). The library-visibility filter lives in exactly ONE place — `listLibraryForUser` — so shelf, search, stragglers, and CSV export stay consistent. Discard/undo reuse the existing `useTrackingMutations` guard + stale-undo + UNDO-toast machinery (mirror the un-own path). Sync stays additive (FR-33) but must not flip owned / upgrade a discarded row.

**Ask First:** Whether to let `import`-kind stragglers be discarded too (they are staging rows, not games — a different `deleteStraggler` semantic). Default: NO — Discard shows only on `unenriched` rows.

**Never:** No hard delete. No reveal-pill / browse page for discarded games (re-add is the only way back). No `archived_on` timestamp — a boolean is enough (add a timestamp only if an audit need appears). Do NOT touch the deferred "Final Fantasy exact-title add" bug or the (retracted) add-confirmation item.

## I/O & Edge-Case Matrix

| Scenario | Input / State | Expected Output / Behavior | Error Handling |
|----------|--------------|---------------------------|----------------|
| Discard a tracked game | `PATCH /api/games/:id/discard {discarded:true}` | flag set; game leaves shelf, search, stragglers, CSV export; 200 `{discarded:true}` | 404 if user has no tracking row for the game |
| Undo discard | UNDO toast activated in window | `{discarded:false}`; card returns to shelf | Stale-undo expires if a newer write on the game occurred (existing guard) |
| Sync sees a discarded PSN game | `discarded:true` + a PSN purchase entry present | owned-flip and membership-upgrade skipped; game stays hidden, not resurrected | n/a |
| Re-add a discarded game by name | add-by-name Save; dup resolves (IGDB link or normalized title) to a row whose tracking is discarded | flag cleared (revived); returns `duplicate` + gameId; client invalidates shelf then opens its detail; card reappears | n/a |
| Discard an `import`-kind straggler | import staging row | Discard control NOT rendered (out of scope) | n/a |

## Design Notes

Single-filter rationale: `loadLibrary` (shelf, search, CSV export) and `listStragglerView` read through `listLibraryForUser`, but `sync` reads `listTrackingForUser` (unfiltered). So one `discarded=false` term in `listLibraryForUser` hides discarded rows everywhere the user browses while sync retains visibility to enforce the tombstone. `setDiscarded` is UPDATE-only (never an upsert) — a missing row must 404, not create an empty tombstone.
