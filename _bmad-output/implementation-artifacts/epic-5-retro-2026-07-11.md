## What went well

- **The Epic 4 lesson landed pre-merge.** 5.1's one high finding — an empty 200 catalog response wiped *every* flag — is the same "Sony returns a valid-looking 200 that means nothing" family as Epic 4's production bug. This time it was caught in dev-time review, patched (0-product 200 now fails as a `provider` error, never runs the clear pass), with a hazard test asserting a flagged game survives an empty catalog. Epic 4 shipped its version to production; Epic 5 caught its sibling before merge.
- **Probe-before-map turned an unknown into a pinned contract.** Epic 4 flagged the PS+ Extra catalog as an undocumented endpoint needing a live probe. 5.1 shipped with the real persisted-query hash, category id, region header, and offset-by-actual-count pagination — modeled from captured behavior, not convention.
- **Zero-duplication trigger seam.** Button and cron run the identical `runPsPlusCheck`; the failed-refresh flag clear and the `psplus_refreshed_at` stamp both live in the shared success path, so 5.2 and 5.3 added surfaces without touching 5.1's flag-diff logic. Cron reads the same persisted `psn_region` — no button/cron divergence by construction.
- **Failed-refresh banner designed against dead-ends.** Review caught that a `no-region` config gap would light an un-clearable banner; fixed so only a retryable `provider` failure lights it. Bookkeeping writes (clear-flag + stamp) wrapped so a post-flag write failure can't flip a genuine refresh to "failed."

## Challenges

1. **The empty-response guard was primed-reviewer luck, not a rule.** The flag-wipe was caught because reviewers carried the Epic 4 wire-contract scar. Nothing in the toolchain *required* an external-data→automated-write path to guard its degenerate/empty response. The next non-PSN provider surface is where that vigilance could lapse. → codified as a standing rule (action item 1, done).
2. **The re-homed follow-up-review gate is still unproven.** All three specs carry `followup_review_recommended: false`, so the gate re-homed in Epic 4 never fired — but Epic 5 ran in a **worktree before the re-home was in the flow**, and Epic 6 did too, so this is a timing artifact, not the gate being theater. **Epic 7 (post-v1.0.0) is its first genuine exercise.** A high-severity finding in 5.1 that still shipped `false` underlines the open question for Epic 7: should a HIGH inline finding force the gate regardless of the author's stamp?

## Significant discovery — external-data guards are still per-story luck

Two epics running, the highest-severity finding has been the same shape: an external provider returns a syntactically valid response that semantically means "nothing" (Epic 4: HTTP 200 + GraphQL denial; Epic 5: HTTP 200 + empty product list), and naive code treats it as ground truth — wiping state or missing a re-auth. Epic 5's was caught pre-merge only because reviewers were primed. `IgdbProvider` (add-by-name, Epic 6) is the third such surface, and Epic 7 will add PS+ catalog-browse fetches — the guard needs to be a standing invariant enforced by the toolchain, not a reflex that depends on who reviewed.

**Resolution (done this session):** codified the **DEGENERATE-RESPONSE GUARD RULE** into `bmad-dev-auto.toml` as a persistent fact beside PROBE-BEFORE-YOU-MAP — every external-data→automated-write path must fail closed on the empty/valid-but-nothing response, with a captured-payload hazard test asserting existing state survives it. IGDB failure-mode probing (action item 2) carries forward to verify Epic 6's IGDB integration against real shapes when it merges.
