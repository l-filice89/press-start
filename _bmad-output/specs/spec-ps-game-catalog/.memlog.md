- (capability) CAP-1 Computed state model: single mutable play status + immutable milestone dates; everything else derived (effective/released/wishlisted/playable-now). Success: every game resolves to an effective state, invariant 'play status OR >=1 milestone' never violated, platinumed game never matches Playing pill. FR-1..14.
- (capability) CAP-2 The Shelf + detail view: one dark cover-forward landing page; cards flip to full editable detail. Success: default shelf = live-status backlog ordered Playing>Paused>Up next>Not started (alpha within), completed/dropped hidden one pill away; milestone edits require confirmation modal. FR-15..19.
- (capability) CAP-3 Filter & search: state/genre/flag filters, OR-within/AND-across, reveal pills; always-available name search over whole library ignoring filters/hidden states. Success: 'completed only' is one pill; search finds any game regardless of active view. FR-19..22.
- (capability) CAP-4 Seed import (one-time): Notion CSV + PS export, title reconciliation across both + games DB, Notion-status mapping, honor CSV Owned, exclude membership claims, enrich genres/cover/dates from games DB. Success: mappable rows land enriched; unmappable/ambiguous go to visible in-app stragglers list (never guessed); skipped-claim count reported. FR-26..32.
- (capability) CAP-5 PS library sync (button): append-only to user data; creates owned/digital games, flips Owned true, skips membership claims, captures cover+store URL. Success: user status/dates/milestones/genres survive every sync; each sync ends with summary (added, flips, skipped claims, needs-attention). FR-33..37.
- (capability) CAP-6 PS+ Extra awareness: flag tracked non-owned games in the user's region Extra catalog, via button + monthly scheduled job, bidirectional; flag hidden once owned. Success: shelf shows 'PS+ catalog as of {date}', failed scheduled refresh surfaces on next open. FR-38..40.
- (capability) CAP-7 Add-by-name: search games DB + existing library, review editable pre-filled data, save; name-only fallback when DB unreachable. Success: discovery-moment add takes seconds and never depends on a third party being up; picking an existing game opens its detail, no duplicate. FR-41..43.
- (capability) CAP-8 Lifecycle date capture: silently auto-record wishlisted/bought/started/completed/platinum on every transition, write-once, manually correctable. Success: unreconstructable dates captured in v1 though nothing consumes them yet; no automatic flow overwrites a recorded date. FR-44..45.
- (capability) CAP-9 PWA + auth + multi-user seam: installable responsive PWA, better-auth magic link, all tracking data scoped to a user id from day one. Success: home-screen icon reaches wishlist-add in seconds on phone, desktop shelf equally first-class, data user-scoped without building sharing/roles. FR-46..48.
- (capability) CAP-10 CSV export (durability): full-library CSV export in v1. Success: games/statuses/milestones/dates/genres/ownership downloadable as CSV — a copy beyond the DB provider's backups. FR-49.
- (constraint) Free-tier hosting is a hard constraint: app is stateless, data in an externally managed DB, and the PS+ scheduled job must also fit the free tier. Free hosting outranks the SQLite preference. NFR-1/2.
- (constraint) Nothing external on render: covers and store links serve from persisted data; third-party APIs are hit only at import/sync/refresh/add time. NFR-3.
- (constraint) Failures surface, never silently retry: expired PS cookie shows refresh instructions (no retry on 401/403); failed external lookup lands the game in the stragglers list. NFR-4/FR-36.
- (constraint) Owned means purchased. Membership-sourced (PS+ claim) entries never create games and never set Owned, in import or sync; when the entitlement source is ambiguous, prefer skipping over flipping Owned (a missed flip is one toggle; a wrong Owned poisons the wishlist). FR-9/addendum. (~123/175 of the real export are claims.)
- (constraint) Append-only to user data: no sync/import/refresh ever modifies or deletes user-entered status, milestones, dates, or genres. Lifecycle dates and milestones are write-once through automatic flows (first value stands), editable only deliberately in the detail view. FR-33/45/6.
- (constraint) Anything computable is computed: derived states (effective/released/wishlisted/playable-now) are never stored or user-set — a manual state can drift, a derived one is always right. PRD section 2 governing principle.
- (constraint) Genres come exclusively from the third-party games DB; Notion's Category column is dropped at import and every game re-tagged via external lookup for a consistent vocabulary. FR-23.
- (constraint) PS auth = pdccws_p session cookie stored in a settings table, editable from the UI, read fresh per call, no retry on failure. FR-36.
- (note) Non-goals (PRD section 6): multi-user & public release (door left open via user-id seam only); non-PlayStation platforms; auto-adding PS+ catalog games to library; automating anything Sony's API can't give reliably; gamification (XP/streaks/badges) ever; personal ratings (Notion Rating not imported, no rating field); genre merge/rename tool (not v1). v1.x-not-v1: trophy sync, critic/user scores, 'leaving PS+ soon' warnings, Google sign-in. Future: sale detection (addendum sketch), PS+ tier settings, play-next tuning, stats dashboards.
- (decision) Success signal = the metric that matters: Notion DB gets archived and never reopened. Demonstrable when (1) every owned game is present without manual entry, (2) a status change takes seconds, (3) wishlist incl. PS+ Extra check is consulted before a purchase, (4) backlog filtering beats Notion's views. Counter-signal: app stops being consulted for what-to-play-next, or wishlist adds regress to batches. PRD section 1.
