# Standing rules — core

Flow-agnostic engineering rules. Apply to any BMAD dev flow (bmad-dev-auto, bmad-dev-story, bmad-quick-dev). Part of https://github.com/l-filice89/bmad-standing-rules — each rule was born from a real production incident; the scar stays in the wording because it is the rationale.

- HAZARD-TEST RULE: if an acceptance criterion names a specific hazard, invariant, or failure mode to guard against (e.g. "API routes must be checked before the SPA fallback", "membership-sourced entries are excluded, count reported", "started_on stamped once, write-once"), the story is not done until a test exists that asserts exactly that behavior — red-then-green, not just "the suite passes". This was written after three early production incidents (a shadowed-route bug, a seed-import exclusion bug, and a missing-secret deploy bug) each shipped past a review pass despite the exact hazard being named in the spec or raised live. Passing lint/typecheck/the existing suite is not sufficient evidence the named hazard is covered — check for it explicitly during implementation and review.

- E2E-COVERAGE RULE: an AC has a UI user flow when a user could observe or drive the behavior through the rendered app. Every AC with a UI flow ships with an end-to-end test (in the project's e2e framework) in the SAME story — a test that asserts that AC's behavior and runs green in the story's verification. Every other AC gets a row in the project's e2e coverage map (a committed file keyed by AC) in the same story, whose reason names what blocks it (no UI flow / unreachable until <feature X>) or what covers it instead; a story with no UI-facing ACs at all may add one summary line. A story that makes a previously-unreachable flow reachable converts the matching skip rows into tests in that same story. This rule and the HAZARD-TEST RULE apply independently: a unit-level hazard test does not exempt a UI-flow AC from e2e coverage, nor vice versa. If the e2e foundation itself is broken, HALT blocked citing the infra failure — never write a false skip reason. Definition of done: EVERY AC in the story has either a passing e2e test or a coverage-map row — audit that map during implementation and review. If the project has no e2e framework yet, standing one up is an early epic, not an exemption.

- PROBE-BEFORE-YOU-MAP RULE: any story touching an external provider's error/failure handling starts with a live probe of the real failure modes — do not model the wire contract from HTTP convention or a happy-path reference script. The origin bug shipped to production because expired-cookie handling was keyed off 401/403 by convention, while the real provider answered 200 + an in-body "Access denied" error. Hazard fixtures for these paths use CAPTURED real payloads (dated, verbatim), never hand-written ones — a stub built from the assumption verifies the assumption, not reality. Applies during implementation and review for any adapter/provider error branch.

- DEGENERATE-RESPONSE GUARD RULE: any external-data → automated-write path must explicitly handle the DEGENERATE response — syntactically valid, semantically empty/nothing — before it drives a write, with a hazard test built from a CAPTURED payload asserting existing state SURVIVES it. Two epics running, the highest-severity finding was this exact shape: one provider answered an expired credential with HTTP 200 + an in-body denial (shipped to production), and another answered with an empty-but-200 list that would have wiped every derived flag (caught in review only because reviewers were primed by the first). "The fetch succeeded" is not "the data is real": an empty list, a 200 carrying an error body, a null/missing collection must fail closed (skip the write, surface a retryable error), never be treated as authoritative emptiness that clears state. Sibling of PROBE-BEFORE-YOU-MAP: probe finds the real shapes, this rule guards the degenerate one. Check every provider→write path during implementation and review; the guard is a rule, not a primed-reviewer reflex.

- SAMPLE-OF-ONE RULE: a spec may NOT assert a constant, enum value, or distribution that ranges over a POPULATION from a single observed instance. Either probe the population and record the real distribution (dated, in the spec), or carry the value as a per-item field read from the data. The origin near-miss: a spec pinned an API service-name constant from ONE sample; a live probe run only because a reviewer challenged it measured 94 of 137 real records using a DIFFERENT value, and the wrong name answered 404 — most of a real user's records would have been silently mis-written into a write-once column, permanently unrecoverable. Sibling of PROBE-BEFORE-YOU-MAP, which it extends from failure modes to population shape: probing one happy-path response tells you the SHAPE of a record, never the DISTRIBUTION across records. At spec/implementation time, for every constant a spec asserts about external data, ask: how many instances was this observed on, and does it vary per item? An unprobed distribution is carried as a field or blocked, never as a constant.

- RESOURCE-BUDGET RULE: any spec claiming a resource budget (a platform's per-request call ceiling, chunk sizes, fan-out limits, rate limits, memory/time caps) must ENUMERATE every consumer with the arithmetic written out — external fetches AND database/binding calls AND middleware's own reads — not just the ones the story is thinking about. The origin project shipped this bug twice on Cloudflare Workers' 50-subrequest ceiling: one budget counted 4 fetches and ignored ~137 sequential D1 UPDATEs (binding calls counted toward the limit; the mock could not see them, so every test passed), and a second ignored the auth middleware's own reads. A budget a mock cannot observe is not a budget — it is a hope. during implementation and review, restate the arithmetic honestly against the real runtime's accounting, and prefer a batched write over a per-row loop on any path whose row count scales with the data.

- TEST-THE-BYPASS-NOT-JUST-THE-REFUSAL RULE: when a change adds a GUARD (a lock, an auth check, a rate limit, a validation boundary), enumerate every code path that reaches the guarded resource and test the BYPASS path, not only the refusal path. A guard whose bypass path has no test is not a guard. The origin: a single-flight lock reached review with the refusal path tested and green (a second sync gets a 409) while the CONTINUATION path — a chunked client loop re-entering the same route with different parameters — had ZERO tests. That was precisely the path that was bypassable, and precisely why the test suite could not see it: both reviewers found it, no test would have. During implementation, for each guard, list the entrypoints (a second route, a retry, a resumed loop, a different parameter shape) and write the test that tries to GET THROUGH; at review, if a guard's entrypoint list is not covered, that is a finding, not a nit. Sibling of the HAZARD-TEST RULE: that one demands a test for a named hazard, this one demands a test for the way around the fix.

- CAPABILITY-IS-NOT-AN-IDENTIFIER RULE: a value the SERVER published to the client — a row id, a cursor, a page token, a filename, a sequence number — is never proof of ownership, permission, or authorship. If a request must prove that it is the one who started an operation, it presents a CAPABILITY the server minted for that purpose: unguessable (a uuid/random token), checked against stored state in the same statement that acts on it, and ideally rotated on each use so a replay of a spent one fails. The origin: a lock keyed its cross-request continuation on a pagination CURSOR — but the cursor was an id the endpoint printed in its own response body, so `?cursor=anything` would have overwritten a RUNNING operation's lock, fanned out to the external provider beside it, then deleted the lock on the way out. The fix was a rotating token, not a better cursor. At spec/implementation time, for any value a request presents as authority, ask: did the server mint this as a secret, or merely hand it back as data? If the latter, it authorizes nothing.

- UI-MOCK-GATE RULE: any story or spec whose acceptance criteria touch a RENDERED surface must carry a lightweight placement mock — ASCII layout, annotated sketch, or throwaway HTML snippet, whichever is fastest — showing WHERE each new element sits on every affected surface, with the product owner's recorded sign-off BEFORE implementation starts. Placement-level only, never pixel-level: the mock pins position and hierarchy; behavior stays in the ACs. Written after an epic shipped every feature behaviorally correct and then spent four layout-churn commits plus an entire follow-on spec chasing where things SIT — specs pin behavior precisely and presentation loosely, and that gap gets closed by the most expensive feedback loop available (build → react → rework) when a minutes-cheap mock closes it before code. At spec time, a UI-touching spec with no signed-off mock is a BLOCK, not a nit — same posture as EXTERNAL-RISK-FLAG. Keep the mock small enough that this never becomes a design-document stage; if drawing it takes longer than reading the spec, it is too detailed.

- EXTERNAL-RISK-FLAG RULE: any epic, story, or spec that touches a third-party service MUST carry an explicit, named risk assessment covering (a) terms-of-service breach, (b) exposure of a real account or credential belonging to the product owner, and (c) legal liability — surfaced to the product owner for recorded sign-off BEFORE any code is written. 'Touches a third-party service' includes: sending any credential or account identifier on the wire, calling undocumented/reverse-engineered endpoints, impersonating an official client, and scraping. The origin: two epics built a credentialed sync impersonating a vendor's mobile app against undocumented endpoints — the ToS risk was known, researched, stated in plain text in the specs, and yet never had a checkpoint that forced the accept/reject conversation; the vendor locked the account, and a cleanup epic deleted two epics of shipped work in a day. The lesson: a known risk without a named checkpoint is an unflagged bet. Anonymous, documented, credential-free calls need only a one-line 'anonymous surface, no account identity' statement. At spec time, if the spec touches an external service and carries no risk flag, that is a BLOCK, not a nit — irreversible external risk gets the same human pause as irreversible data loss.

- WCAG-COMPLIANCE NUDGE: the app must remain WCAG compliant, and any spec introducing or reworking an INTERACTIVE widget (menu, dialog, tab set, radio group, toggle, combobox) names the ARIA pattern it follows (WAI-ARIA Authoring Practices) in the spec, so roles, keyboard operability, and focus order are designed up front rather than retrofitted in review. A NUDGE, not a block — unlike UI-MOCK-GATE there is no sign-off step. Written after two consecutive epics shipped a11y-shaped review patches: widgets landed mouse-only and ARIA-invalid (plain buttons inside a role=menu, outside the roving focus) and forced a mid-review rework of shared focus machinery — the exact rework class an up-front pattern name prevents. During implementation and review, check each new interactive element against its named pattern: correct roles, keyboard reachability, focus management, and state conveyed by words/pressed-state, never color alone.

- PRESERVE-VS-CLEAR RULE: any partial-update write path (rematch, refresh, merge, upsert) must state PER FIELD whether an ABSENT value in the incoming data means PRESERVE existing state or CLEAR it — absence and explicit null are different signals and the spec says which is which. And any consumer of an external endpoint that can answer `200 []` (or 200 + empty/missing collection) must state, per endpoint, whether empty is a FAILURE (fail closed, skip the write) or LEGITIMATE ABSENCE (write the clear). Each ruling carries a hazard test exercising the absent/empty case. The origin epic's hazard class: 4 of 12 medium findings were "absent conflated with explicit null" — a rematch wiping scores, a rematch inheriting stale data forever (found only in follow-up review), a hollow-200 clearing real dates — and one story shipped a permanently-lit, unclearable banner into review because the empty-reply direction was GUESSED fail-closed when empty was in fact normal absence. Extends DEGENERATE-RESPONSE GUARD: that rule demands the empty reply be handled; this one demands the spec rule WHICH WAY, per field and per endpoint, before code. At spec time, a spec with a partial-update write or a 200-[]-capable read that lacks these rulings is a finding against the SPEC; during implementation and review, verify each ruling has its hazard test.

- STALE-READ-VALIDATOR RULE: when a change introduces or reworks a conditional-read architecture (ETag/304, version stamps, generation counters, any cache validator) over SHARED data, enumerate every WRITER of that data — not just the acting user's own path — and verify each write bumps the validator every affected reader checks; then test staleness PER WRITER CATEGORY, not just the cache-hit/refusal path. The origin epic turned per-user reads into SQL derivations over shared snapshots and minted a five-finding defect class inside the same epic: a shared fact was written but only the WRITING user's version bumped, so every co-observing user's client 304'd against stale data forever — two HIGHs, a post-"done" follow-up HIGH, and two mediums all shared this one shape. Sibling of TEST-THE-BYPASS-NOT-JUST-THE-REFUSAL: that rule tests the way AROUND a guard; this one tests the way stale data SURVIVES a cache check. When the read model changes, enumerate the writers, not just the readers.
