# Standing rules — epic process

Rules that depend on epic/story machinery (merge gates, followup_review_recommended stamps, cross-story delegation, deferred-work ledger). Apply to bmad-dev-auto and bmad-dev-story; meaningless in bmad-quick-dev. Same repo and rationale as standing-rules-core.md.

- FOLLOW-UP-REVIEW CONTRACT: a story finishing with followup_review_recommended: true does NOT reach the epic's merge gate until an independent review pass runs over its change (second-opinion angles + verify) OR a decline is recorded with reason. The origin: this gate once lived in an orchestrator policy file and died silently when that tool was retired — three stamps sat unconsumed and a named hazard shipped. It lives in this customization file because prompt-level facts survive tooling changes. Check the story's followup_review_recommended flag at review; if set, run or record the independent pass before marking the epic mergeable.

- FOLLOW-UP-REVIEW AUTO-FORCE ON HIGH: a HIGH-severity inline review finding on a story FORCES the independent follow-up pass above, regardless of the author's followup_review_recommended stamp — and unlike the self-stamp path it CANNOT be declined away. Extends the FOLLOW-UP-REVIEW CONTRACT: that gate consumes the author's self-assessment; this one adds a trigger the author cannot clear. Rationale: the self-stamp gate fires and works, but a self-stamp is still author discretion, and a HIGH finding is exactly the severity where a second opinion must not be optional. At review, if ANY inline finding is HIGH, treat followup_review_recommended as true and run the independent pass before marking the epic mergeable — no decline branch for HIGH.

- DELEGATED-WORK-CARRIES-AN-AC RULE: when a story delegates work to another story — 'X owns that loop', 'lands with story N', a residual-risk line naming a later owner — the RECEIVING story's spec MUST carry at least one acceptance criterion that verifies the delegated work was actually done. Prose in a run result, a residual-risks note, or a code comment is NOT a carrier: it has no gate, no test, and no owner-of-record. The origin: one story's result recorded 'no client drives the sweep yet — the next story owns that loop', the next story shipped only its own read endpoints, every reviewer did their job (the gap was in no story's diff), and a whole data column stayed empty in every environment until the product owner noticed dead UI days later. At spec time, when planning a story, grep the epic's earlier specs (run results, residual risks, review triage) for delegations naming this story and convert each into an AC; at review, a delegation whose receiving story lacks the AC is a finding against the SPEC, not a nit.

- POST-EPIC OPERATOR SWEEP RULE: after an epic's LAST story completes and before its merge gate, a mandated OPERATOR-PERSPECTIVE pass runs over the epic as a running system, not a diff: (a) triage every open deferred-work ledger entry the epic created or touched — each one closed, fixed, or explicitly discarded with reason; (b) audit the epic's TEMPORAL behavior — every scheduled/recurring path (cron cadence, retry/backoff, quarantine, window resets) answered against 'what does month three look like', including failure recovery with no user action available; (c) re-state the combined resource budgets across ALL scheduled jobs and routes the epic ships (RESOURCE-BUDGET arithmetic) and split or re-schedule anything at zero headroom. The origin: reviews caught 19 HIGHs in diffs while a refresh cadence bug, an all-users failure flag, an orphaned delegation, and a zero-headroom request ledger were all raised by the PRODUCT OWNER post-epic — every item was ledgered (visible), none was prioritized (no owner, no trigger). Reviews audit diffs; nobody audits time. This pass is the owner. At the epic's end, an unswept ledger or an unaudited scheduled path is a BLOCK on the merge gate, same posture as the FOLLOW-UP-REVIEW CONTRACT.
